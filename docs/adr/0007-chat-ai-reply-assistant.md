# ADR 0007：聊天内 AI 助手回复（可迭代改写 + 用户级 prompt 设置）

- 状态：提议
- 日期：2026-07-08
- 决策者：平台核心组（待评审）

## 背景
扩展需求要求在私聊和群聊中提供一个低摩擦的 AI 回复助手。第一版目标不是“自动发消息”，而是：
- 一键/快捷键唤起 AI
- 基于当前会话上下文生成回复草稿
- 用户可以修改本次提示词，并让 AI 基于当前草稿继续改写
- 用户可在设置中保存自己的 system prompt，控制回复风格

这个能力会同时影响聊天主流程、用户资料契约和 AI 调用提示词结构，因此需要单独记录决策。

## 决策
1. 新增聊天 AI 端点：
   ```yaml
   POST /v1/messages/ai-reply
   ```
   请求参数包含：
   - `scope`: `FRIEND | TEAM`
   - `peerId`: 会话对端用户 ID 或 teamId
   - `draftText`（可选）: 当前草稿；为空时表示首次生成
   - `instruction`（可选）: 本次修改要求；为空时表示按默认风格生成/改写

   响应返回：
   - `suggestion`: AI 生成的建议回复文本
   - `contextCount`: 实际使用的上下文消息条数

2. 扩展用户资料契约，新增用户级 AI 设置：
   ```yaml
   aiSettings:
     systemPrompt: string
   ```
   第一版只提供 `systemPrompt`，由前端在聊天页里配合“本次修改要求”使用。

3. 第一版不新增数据库表，用户 AI 设置先复用 `user.privacy_settings` JSON 存储，后端兼容解析。

4. AI 回复采用“生成/改写草稿，不自动发送”：
   - 后端返回建议文本
   - 前端写入输入框，由用户决定是否继续改写或发送

5. 上下文范围限定为“当前用户有权访问的当前会话最近 N 条消息”，严禁跨会话拼接。

## 后果（好处 / 代价 / 影响谁）
- 好处：
  - AI 不只是一锤子生成，用户可以围绕现有草稿持续润色。
  - 不自动发消息，降低误发和越权风险。
  - 用户可自定义回复风格，体验更贴近真实聊天。
- 代价：
  - 需要新增 OpenAPI 端点和用户资料字段语义。
  - AI 提示词拼装更复杂，需要明确区分“上下文 / 草稿 / 本次修改要求”。
- 影响谁：
  - 后端 `module/social`、`module/user`
  - 前端 `ChatView`、`Profile`
  - 契约维护由平台核心组审批

## 备选方案
1. 让 AI 自动发送消息：交互更短，但误发风险高，否决。
2. 只保留 system prompt，不支持本次改写要求：实现更快，但用户无法围绕草稿持续调整，否决。
3. 为每次 AI 会话单独建表保存多轮上下文：可扩展性更好，但开发面过大，不适合本次增量，否决。
